# LL-002 — La doble pregunta: producto + ecosistema

| | |
|---|---|
| **Fecha** | Junio 2026 |
| **Origen** | Stay — cierre de 09 MVP Scope / entrada a Build Specs |
| **Autor** | Founder |

## Observación

A partir de cierto punto del recorrido, dejamos de **diseñar Stay** y empezamos a **usar Stay
como el primer producto que valida el RUN72 Product Operating System**. Eso cambia el criterio
de evaluación de cada decisión.

## Principio

Toda decisión, de acá en adelante, se evalúa con **dos** preguntas (no una):

1. **¿Es correcto para el producto?** (¿sirve a Stay / a la hipótesis del MVP?)
2. **¿Hace más fuerte al ecosistema RUN72?** (¿deja un componente, contrato, patrón o aprendizaje
   reutilizable por el 4º, 5º y 6º producto?)

Si una decisión es buena para el producto pero no aporta al ecosistema, está bien — pero si
**puede** aportar al ecosistema con poco esfuerzo extra, ese esfuerzo casi siempre vale.

## Consecuencia

- Acompaña todas las etapas siguientes (Build Specs, Sprints, Arquitectura, Roadmap).
- Refuerza la Regla del Tres y el Capability Catalog: lo que se reutiliza se promueve.
- Es la operacionalización diaria de la tesis RUN72 (ver LL-001).
